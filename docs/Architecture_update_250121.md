This document is the source-of-truth description of Elevationary’s newsletter business architecture:

- Lead-gen workflow (Instantly) and conversion (Stripe)

## 2. Core idea (high-level)

Elevationary publishes multiple niche newsletters (department-focused) generated by AI.

- Emails contain short summaries designed to create curiosity.
- Links drive traffic to Elevationary properties.
- Viewing “richer stories” (expanded summaries + archive) requires a paid subscription.
- The system is designed for maximum automation with phased manual review early on.

## 6. End-to-end workflows

### 6.1 Phase 0 — Seed distribution to known contacts (“cool leads”)

Input: ~2,500 existing contacts
Steps:

1) Upload/enrich contacts in Instantly (email validation + enrichment as needed)
2) Send personal-sounding invitation to free newsletter subscription
3) If accepted → move to rich-text newsletter delivery flow
4) If no response after 4 attempts → pause and recycle slot later

### 6.2 Phase 1 — Cold acquisition at scale via domain rotation

Objective: protect primary domain reputation by using warmed secondary domains
Infrastructure:

- 20 outreach domains (one per newsletter)
- 5 warmed email inboxes per domain
- Each inbox sends 30 emails/day
  Capacity math:
- 20 * 5 * 30 = 3000 emails/day
- 3000/day * 5 days/week * 4 weeks = 60,000 emails/month

Outreach logic:

- Prospects receive plain-text sequence offering free subscription.
- On acceptance, stop plain-text outreach and transition to newsletter delivery.

### 6.3 Conversion — Freemium to paid

Trigger: recipient clicks “read richer story” link in newsletter email (or equivalent)
Flow:

1) User lands on microsite conversion page
2) Stripe checkout processes subscription purchase
3) Post-checkout:
   - user is granted paid entitlement to specific newsletter(s)
   - user gains access to richer stories + archive for those newsletters
4) Delivery: paid subscriber starts receiving rich-text newsletters next day

## 7. Microsite purpose and boundary

Reason microsite exists:

- Public site is on Google Sites (fast + simple but limited)
- Microsite provides:
  - reliable Stripe integration
  - conversion pages that appear “on brand”
  - gated content (richer stories + archive)
  - security controls that Google Sites can’t provide
  - infrastructure for future agentic commerce protocols (ACP/UCP)

User journey:
Public site (Google Sites) → “Subscribe / Learn more” → Microsite (Cloudflare Pages) → Stripe checkout → entitlement + gated content

## 13. Risks & mitigations

- Deliverability risk: domain rotation + warmed inboxes
- Compliance risk: suppression list hygiene + unsubscribe plumbing
- Hallucination risk in AI-generated content: staged manual review + source linking
- Paywall complexity risk: choose simplest entitlement mechanism first (Stripe-first)
- Tool sprawl risk: enforce source-of-truth rules in this document
